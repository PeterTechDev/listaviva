# PWA Offline Caching — Design Spec

**Goal:** Cache visited catalog and provider pages so users can browse previously viewed content with poor or no connectivity.

**Approach:** Config-only — extend the existing `@ducanh2912/next-pwa` setup in `next.config.ts`. No custom `sw.ts`. No new packages.

---

## Architecture

Three files touched:

| File | Change |
|---|---|
| `next.config.ts` | Extend `withPWA` config with caching rules, fallback, and front-end nav caching |
| `src/app/offline/page.tsx` | New root-level static offline fallback page |
| `public/manifest.json` | Update `theme_color` to match current design token (`#C85C38`) |

The service worker is fully generated by `@ducanh2912/next-pwa` (Workbox) at build time. It is disabled in development (`disable: process.env.NODE_ENV === "development"` — keep as-is).

---

## Caching Strategy

### App shell (automatic precaching)
JS bundles, CSS, fonts, and static assets under `/_next/static/` are automatically precached by Workbox. No explicit rule needed.

### Page routes (runtime caching — StaleWhileRevalidate)
Serve from cache immediately if available; fetch and update cache in the background.

URL patterns:
- `/pt-BR` and `/en` (home pages)
- `/*/categories`
- `/*/category/*`
- `/*/provider/*`

Config: max 50 cached pages, 30-day TTL, 5 MB storage quota.

### Images (runtime caching — CacheFirst)
Serve from cache; only hit network on a miss.

URL patterns:
- `/_next/image*` (Next.js image optimization)
- Supabase Storage CDN URLs for provider photos

Config: max 50 entries, 30-day TTL.

### Client-side navigation caching
Set `cacheOnFrontEndNav: true` so SPA navigations (tapping category cards, tapping provider cards via the Next.js router) also populate the runtime cache. Without this, only hard navigations and reloads would be cached — missing most of how users browse Listaviva.

## Offline Fallback Page

**Location:** `src/app/offline/page.tsx` — root-level, outside `[locale]`, not subject to next-intl middleware.

**Content:**
- App name and tagline
- "Você está sem conexão" message
- "Tente novamente" button calling `window.location.reload()`
- Styled with design tokens: `bg-background` (#FAF6EF), `text-primary` (#1C1410), `bg-accent` (#C85C38)
- No Header component, no Supabase calls, no i18n — must be fully static

**Fallback config:** `fallbacks: { document: "/offline" }` in `withPWA`.

## Manifest Update

`public/manifest.json` `theme_color` is currently `#10b981` (emerald green — leftover from before the redesign). Update to `#C85C38` to match the current accent design token.

## Error Handling

- If the service worker fails to register (e.g., HTTPS not available), the app continues to work normally online — PWA is progressive enhancement.
- If the offline fallback page itself fails to load (e.g., not precached), the browser shows its native offline error — acceptable edge case.
- Cache quota exceeded: Workbox's `maxEntries` and `maxAgeSeconds` expiry prevent unbounded growth.

## Testing

All tests are manual via Chrome DevTools (service workers cannot be meaningfully unit tested).

**Setup:** `npm run build && npm start` (service worker is disabled in dev).

**Test 1 — App shell offline:**
1. Load the app, confirm SW registered (DevTools → Application → Service Workers)
2. Set Network → Offline
3. Reload the page → shell should load from cache

**Test 2 — Cached page accessible offline:**
1. Online: visit a category page and two provider profiles
2. Set Network → Offline
3. Navigate to those same pages → should load from cache

**Test 3 — Uncached page shows fallback:**
1. Offline: navigate to a page not previously visited
2. Should show the `/offline` fallback page (not a browser error)

**Test 4 — SPA navigation caching:**
1. Online: open the home page, tap a category (client-side nav), tap a provider (client-side nav)
2. Set Network → Offline
3. Navigate back to those pages → should load from cache (validates `cacheOnFrontEndNav`)
